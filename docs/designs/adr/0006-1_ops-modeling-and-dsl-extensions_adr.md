# ADR: 决策：运维实体建模与DSL扩展方案 (OpsModelingAndDslExtensionsDecision)

## 1. 决策背景 (Context)

在规划“运维基础组件与DSL扩展”（见RFC-0006）的任务时，我们面临一系列关键的架构选择。这些选择将直接影响新功能的灵活性、可维护性以及与现有 `Matrix` 框架的兼容性。我们需要明确如何表示运维实体、如何处理动静态数据、以及如何以最优雅的方式实现配置复用和多视角建模。

本ADR旨在正式地、永久地记录我们关于此问题的最终决策及其原因。

## 2. 架构决策 (Decision)

**我们决定采用一套分层、解耦的方案：通过创建新的 `types.Node` 组件来表示运行时运维实体，同时扩展DSL以分离执行流 (`connections`) 与逻辑拓扑 (`relations`)，并引入 `imports` 机制来实现配置的模块化复用。**

## 3. 决策理由 (Rationale)

此决策是在对多种备选方案进行详细讨论和权衡后做出的，旨在实现功能、性能和可维护性之间的最佳平衡。

### 3.1. 决策点一：如何表示运维实体？ (HowToRepresentOpsEntities)

*   **备选方案A**: 使用静态知识图谱节点（`.md` 文件）来定义运维实体，并通过一个专门的服务来解析和可视化它们。
*   **备选方案B**: 创建可被规则链实例化的、实现了 `types.Node` 接口的运行时组件。
*   **最终决策**: **选择方案B**。
*   **理由**: 方案A将运维实体隔离在 `Matrix` 运行时之外，使其无法参与到自动化的执行流程中。方案B将运维实体作为框架的“一等公民”，不仅能被用于拓扑展示，还能直接参与到健康检查、自动化部署等可执行的工作流中，这为未来的AIOps场景提供了无限的可能性。

### 3.2. 决策点二：如何处理节点的动静态属性？ (HowToHandleDynamicAndStaticProperties)

*   **备选方案A**: 将所有属性（如IP地址、操作系统、CPU使用率）都放在节点的 `Configuration` 中。
*   **备选方案B**: 在节点内部对动静态属性进行分离。
*   **最终决策**: **选择方案B**。
*   **理由**: 方案A混淆了配置与状态，导致配置臃肿且难以管理。方案B将不变的**静态配置**（如IP地址）保留在 `Configuration` 中，而将易变的**动态状态**（如CPU信息）维护在节点内部的 `State` 结构中，并通过特定消息（如 `{"action": "PROBE"}`）进行惰性更新。这使得模型更清晰，职责更单一，并实现了状态的按需获取和缓存。

### 3.3. 决策点三：如何同时支持拓扑建模和流程执行？ (HowToSupportTopologyAndExecution)

*   **备选方案A**: 复用现有的 `connections` 数组，通过增加 `type` 字段来区分“执行连接”和“逻辑关联”。
*   **备-选方案B**: 在DSL中增加一个全新的 `relations` 数组，与 `connections` 并行。
*   **最终决策**: **选择方案B**。
*   **理由**: 方案A会污染核心的 `connections` 语义，并给 `Matrix` 运行时带来不必要的复杂性（如需要过滤掉非执行连接）。方案B则实现了完美的**关注点分离**：`connections` 专用于必须是DAG的**执行流**，由 `Matrix` 运行时处理；`relations` 专用于可有环的**逻辑拓扑**，由外部可视化工具处理。这确保了对核心引擎的零影响，同时提供了最大的建模灵活性。

### 3.4. 决策点四：如何实现配置复用？ (HowToAchieveConfigurationReuse)

*   **备选方案A**: 仅使用框架现有的 `SharedNodePool` 机制。
*   **备选方案B**: 引入一个新的 `imports` 关键字到DSL中。
*   **最终决策**: **两者结合，以方案B为主**。
*   **理由**: `SharedNodePool` 解决的是**运行时实例共享**的问题（性能优化），而我们面临的核心问题是**定义时配置复用**（可维护性）。仅使用方案A无法避免在多个工作流文件中重复定义节点。因此，我们决定引入 `imports` 机制，允许一个工作流DSL文件复用一个基础拓扑DSL文件中的节点定义，实现“数据与行为分离”。同时，这个方案与 `SharedNodePool` 完美兼容，可以在此基础上进一步对昂贵资源进行实例共享。

综上所述，我们选择的这套组合方案，是在充分理解现有框架并对多种可能性进行权衡后得出的最优解，它在功能、性能、可维护性和前瞻性上都达到了理想的平衡。
