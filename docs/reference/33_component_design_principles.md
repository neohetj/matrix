# 1. 核心原则一：代码优先，复用为王 (Principle 1: Code First, Reuse is King)

在Matrix中，规则链是业务逻辑的核心载体，而业务节点是构成规则链的原子单元。设计的优劣直接决定了系统的可维护性、可扩展性和整体性能。因此，我们确立“代码优先，复用为王”作为首要设计原则。

**在着手设计任何新的规则链或业务节点之前，必须首先对现有资产进行彻底的搜索和分析。**

## 1.1. 搜索现有资产 (Searching Existing Assets)

开发者有责任使用系统提供的工具（未来将通过MCP工具实现）来回答以下问题：

*   是否存在功能相似或完全相同的业务节点？
*   是否存在可以被稍加改造或扩展以满足新需求的节点？
*   是否存在处理相似业务逻辑的规则链，可以作为参考或模板？

## 1.2. 节点复用与融合策略 (Node Reuse and Fusion Strategy)

*   **优先复用 (Prioritize Reuse)**: 如果找到完全匹配的现有节点，**必须**直接复用。
*   **谨慎融合 (Consider Fusion Carefully)**: 如果发现功能相近的节点，需要评估将其逻辑融合以支持新旧两种业务场景的可能性。
    *   **融合前提**: **绝对保证**融合后的节点不会对任何原有的业务功能产生负面影响。这需要通过完备的回归测试来验证。
    *   **融合方式**: 通常通过为节点增加新的配置参数或扩展其输入消息体来实现。
*   **最后选择：创建新节点 (Create New Node as a Last Resort)**: 只有在确认没有任何可复用或可融合的节点时，才应创建新的业务节点。

# 2. 核心原则二：单一职责与高内聚 (Principle 2: Single Responsibility & High Cohesion)

每个业务节点都应遵循单一职责原则（SRP），即一个节点只做一件事情，并把它做好。

*   **反模式**: 一个节点同时负责数据格式转换、业务逻辑计算和调用外部API。
*   **最佳实践**: 将上述功能拆分为三个独立的节点：`DataTransformNode`、`BusinessLogicNode` 和 `ApiCallNode`。

这样做的好处是：
*   **极高的可复用性**: 每个小节点都可能在其他规则链中被复用。
*   **易于测试**: 可以独立测试每个节点的功能。
*   **清晰的逻辑流**: 规则链本身就能清晰地展示业务处理的步骤。

# 3. 核心原则三：松耦合与配置驱动 (Principle 3: Loose Coupling & Configuration-Driven)

*   **避免硬编码**: 节点内部严禁硬编码任何业务参数、URL、API密钥等。所有可变信息都应通过节点配置或输入消息传入。
*   **面向接口设计**: 节点间的交互应依赖于标准化的消息结构，而不是特定节点的内部实现。

# 4. 核心原则四：渐进式重构与演进 (Principle 4: Progressive Refactoring and Evolution)

我们鼓励“先实现，后优化”的演进式设计哲学。与其在初期就追求完美的原子化拆分，不如先快速构建一个能满足当前业务需求的、功能内聚的节点。

## 4.1. 演进SOP：何时以及如何拆分 (Evolution SOP: When and How to Split)

1.  **首次实现 (Initial Implementation)**: 当一个新功能不涉及明显的、已存在的复用逻辑时，可以创建一个功能相对完整的“粗粒度”节点。**关键是保证其内部逻辑的高内聚性**。
2.  **识别复用信号 (Identifying Reuse Signals)**: 当你或团队其他成员在开发新功能时，发现需要实现“粗粒度”节点中的某一部分逻辑时，这就是一个强烈的重构信号。
3.  **执行拆分 (Executing the Split)**:
    *   将“粗粒度”节点中可被复用的逻辑提取出来，封装成一个新的、独立的“细粒度”节点。
    *   将原有的“粗粒度”节点进行重构，使其调用这个新的“细粒度”节点。
    *   确保所有调用旧节点的规则链都能无缝切换到新模式，通常这意味着保持原有节点的配置和接口不变。
    *   为新旧节点都补充完整的单元测试和集成测试。

这种方法在快速响应业务需求和保持系统长期可维护性之间取得了良好的平衡。

> **问：过度拆分节点是否会导致规则链过于复杂？**
> **答：** 这是一个非常好的问题，也是我们引入“渐进式重构”原则的原因。我们不要求在设计之初就进行完美的原子化拆分。相反，我们允许为了快速交付而创建功能更集成的节点。但系统必须是演进的，当明确的复用需求出现时，我们就必须遵循SOP进行重构，将通用逻辑沉淀为更细粒度的、可复用的节点。这种“先让它工作，再让它更好”的策略，可以有效避免过度设计。

> **给Agent的指令:** 在生成规则链设计方案时，必须首先声明已对现有节点进行了搜索。如果创建了新节点，必须提供理由，说明为何现有节点无法满足需求。生成的每个节点都必须符合单一职责和配置驱动的原则。
